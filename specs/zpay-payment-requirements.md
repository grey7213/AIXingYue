# 惑梦 ZPAY 自动充值 Requirements

Updated: 2026-07-13

## 目标

- 使用 ZPAY 兼容易支付接口替代用户侧“爱发电购买卡密 → 回站兑换”的主充值流程。
- 用户登录后选择固定积分包或月度额度包，选择支付宝/微信，跳转 ZPAY 收银台。
- ZPAY 异步回调验签成功后自动增加当前账号的 `paid_points`，无需手动兑换码。
- 保留现有兑换码库存和兑换 API 作为客服补偿/回滚能力，但默认不再作为主购买入口。

## 官方协议约束

- 网关支付页：`/submit.php`，支持 GET/POST；项目使用服务端生成签名后的 HTTPS 支付 URL。
- 回调：GET `notify_url` / `return_url`，成功状态仅接受 `TRADE_SUCCESS`。
- MD5：参数名按 ASCII 升序；排除 `sign`、`sign_type` 和空值；拼接 `a=b&c=d` 后直接追加商户 KEY，MD5 小写。
- notify 必须在 5 秒内返回纯文本 `success`，重复通知必须幂等。

## 安全要求

- PID/KEY 只存在服务器环境变量，不进入 Git、前端、API 响应或日志。
- 套餐、金额、积分全部由服务端目录决定；客户端只能提交 `plan_id` 与 `pay_type`。
- notify 同时校验 PID、签名、订单号、金额、支付状态和套餐快照。
- 同一订单只允许一次积分入账；并发重复通知使用 SQLite `BEGIN IMMEDIATE` 和条件状态更新防重。
- 支付 URL 只允许 `https://zpayz.cn/`，return/notify URL 固定为本站地址，不接受客户端传入。
- 用户只能查询自己的订单；未登录不得创建或查询订单。

## 套餐与余额

- 沿用线上 4 个积分包和 3 个一次性月度额度包。
- 到账类型统一为 `paid_points`，同时同步兼容总余额 `points`。
- 月度包当前是一次性额度包，不实现自动续费。

## 验收

- 本地测试覆盖签名向量、伪回调、重复回调、错误签名、错误金额、错误 PID、非成功状态和越权查询。
- 线上能创建 ZPAY 待支付订单并打开官方收银台；响应和页面不泄露 KEY。
- 未支付订单不会到账；成功 notify 只入账一次。
- 服务/Nginx active，内外 `/health` OK，`CONTENT_MODE=local_only`。

## 非目标

- 本次不调用退款 API，不做自动退款或自动续费。
- 不删除历史兑换码、兑换记录或爱发电运营库存。
- 不改 APK。
